# Internationalization (i18n) — Diabeo BackOffice

Diabeo BackOffice supports three languages: **French** (default), **English**, and **Arabic** (RTL).
Locale preference is stored client-side in a cookie and synced to user profile in the backend.

---

## Architecture

### Setup

next-intl is configured in Next.js 16 App Router with:

- **Locales**: `fr` (default), `en`, `ar`
- **Locale detection**: Cookie-based with fallback to `defaultLocale`
- **Messages**: JSON files in `messages/{locale}.json`
- **RTL support**: CSS direction control + logical properties for Arabic

### File Structure

```
src/
├── i18n/
│   ├── config.ts          # Locale constants, RTL detector
│   └── request.ts         # Server-side getRequestConfig() for locale resolution
messages/
├── fr.json                # French translations
├── en.json                # English translations
└── ar.json                # Arabic translations (RTL)
```

## Configuration Files

### `src/i18n/config.ts`

```typescript
export const locales = ["fr", "en", "ar"] as const
export type Locale = (typeof locales)[number]

export const defaultLocale: Locale = "fr"

/** RTL locales — used to set dir="rtl" on <html> */
export const rtlLocales: readonly Locale[] = ["ar"] as const

export function isRtlLocale(locale: Locale): boolean {
  return (rtlLocales as readonly string[]).includes(locale)
}

export const LOCALE_COOKIE = "diabeo_locale"
```

Exports:
- `locales` — allowed locale codes
- `defaultLocale` — fallback when cookie is absent or invalid
- `rtlLocales` — languages requiring right-to-left text direction
- `isRtlLocale()` — helper to determine if layout should be RTL
- `LOCALE_COOKIE` — cookie key for locale persistence

### `src/i18n/request.ts`

```typescript
import { getRequestConfig } from "next-intl/server"
import { cookies } from "next/headers"
import { defaultLocale, locales, LOCALE_COOKIE, type Locale } from "./config"

export default getRequestConfig(async () => {
  const cookieStore = await cookies()
  const cookieLocale = cookieStore.get(LOCALE_COOKIE)?.value as Locale | undefined
  const locale = cookieLocale && locales.includes(cookieLocale)
    ? cookieLocale
    : defaultLocale

  return {
    locale,
    messages: (await import(`../../messages/${locale}.json`)).default,
  }
})
```

Reads the locale cookie and loads the corresponding message JSON.
Falls back to `defaultLocale` if cookie is absent or invalid.

## Usage in Components

### Server Components

In server components, use `getTranslations()` from next-intl/server:

```typescript
import { getTranslations } from "next-intl/server"

export default function ServerPage() {
  const t = getTranslations("dashboard")
  
  return (
    <h1>{t("title")}</h1>  // Looks up "dashboard.title" in messages
  )
}
```

### Client Components

In client components, use `useTranslations()` hook:

```typescript
"use client"

import { useTranslations } from "next-intl"

export function ClientButton() {
  const t = useTranslations("common")
  
  return <button>{t("save")}</button>  // Looks up "common.save"
}
```

### Dynamic Parameters

```typescript
const t = useTranslations("validation")
const message = t("required", { field: "Email" })
// "validation.required" = "Field {field} is required"
// Output: "Field Email is required"
```

### Pluralization

```typescript
const t = useTranslations("items")
const label = t("count", { count: 5 })
// "items.count" = "{count, plural, one {# item} other {# items}}"
// Output: "5 items"
```

## RTL (Right-to-Left) Support

### HTML Direction

The layout root sets `dir` based on locale:

```tsx
import { isRtlLocale } from "@/i18n/config"
import { getLocale } from "next-intl/server"

export default async function RootLayout({ children }) {
  const locale = await getLocale()
  
  return (
    <html dir={isRtlLocale(locale) ? "rtl" : "ltr"}>
      {children}
    </html>
  )
}
```

### CSS Logical Properties

Use logical properties instead of directional properties for responsive RTL:

```css
/* Wrong — ignores RTL */
.card {
  margin-left: 1rem;
  padding-right: 2rem;
  border-left: 1px solid;
}

/* Correct — works in both LTR and RTL */
.card {
  margin-inline-start: 1rem;   /* Left in LTR, right in RTL */
  padding-inline-end: 2rem;    /* Right in LTR, left in RTL */
  border-inline-start: 1px;    /* Left border in LTR, right in RTL */
}
```

Common Tailwind logical utilities:
- `ms-*` (margin-inline-start) — left margin in LTR, right in RTL
- `me-*` (margin-inline-end) — right margin in LTR, left in RTL
- `ps-*` (padding-inline-start)
- `pe-*` (padding-inline-end)
- `is-*` (inline-size — width, but direction-aware)
- `block-*` (block-size — height, unchanged)
- `start-*` / `end-*` (positioning)

### Flexbox & Grid

Flexbox automatically reverses row direction in RTL:

```tsx
// Direction automatically flips in RTL
<div className="flex flex-row gap-4">
  <div>First item</div>
  <div>Second item</div>
</div>
```

## Changing Locale at Runtime

Set the locale cookie and optionally reload:

```typescript
"use client"

import { LOCALE_COOKIE, locales } from "@/i18n/config"

export function LocaleSwitcher() {
  const handleLocaleChange = (newLocale: string) => {
    // Set cookie (client-side via JavaScript)
    document.cookie = `${LOCALE_COOKIE}=${newLocale}; path=/; max-age=31536000`
    
    // Reload page to apply new locale
    window.location.reload()
  }
  
  return (
    <select onChange={(e) => handleLocaleChange(e.target.value)}>
      {locales.map(locale => (
        <option key={locale} value={locale}>
          {locale.toUpperCase()}
        </option>
      ))}
    </select>
  )
}
```

Or use the `setUserLocalePreference` API endpoint (Phase 11+) to sync preference to user profile:

```typescript
const response = await fetch("/api/account/locale", {
  method: "PUT",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ locale: "ar" }),
})

if (response.ok) {
  // Set cookie and reload
  document.cookie = `${LOCALE_COOKIE}=ar; path=/; max-age=31536000`
  window.location.reload()
}
```

## Translation Namespaces

To keep message files organized and translation keys grouped by feature, use namespaces:

| Namespace | Purpose | Used In |
|-----------|---------|---------|
| `"common"` | Generic labels (Save, Cancel, Loading, Error, OK) | All components |
| `"period"` | Time period options | PeriodSelector |
| `"nav"` | Navigation item labels | NavigationShell, Sidebar |
| `"clinical"` | Medical terminology (hypo, TIR, CGM, etc.) | Glucose display, charts |
| `"dashboard"` | Dashboard-specific text | Dashboard page |
| `"patients"` | Patient list & detail | Patient pages |
| `"medications"` | Medication management | Medication pages |
| `"analytics"` | Analytics & reporting | Analytics page |
| `"errors"` | Error messages | Form validation, API errors |
| `"validation"` | Form field validation messages | Forms |

## Locale Detection Priority

1. **Cookie** — User-set locale in `diabeo_locale` cookie
2. **User profile** — Synced from backend account preference (Phase 11+)
3. **Browser** — `Accept-Language` header (not currently used; can be added later)
4. **Default** — `defaultLocale` ("fr")

## TypeScript Support

Message keys and parameters are type-checked via next-intl:

```typescript
// ✅ Correct
const t = useTranslations("dashboard")
t("title")

// ❌ TypeScript error — namespace doesn't exist
const t = useTranslations("nonexistent")

// ❌ TypeScript error — key doesn't exist in namespace
t("invalidKey")
```

Enable strict type checking in `tsconfig.json`:

```json
{
  "compilerOptions": {
    "strict": true,
    "noImplicitAny": true
  }
}
```

## Best Practices

1. **Always use translation keys** — Never hardcode user-facing text
2. **Group by feature** — Create separate namespaces for major features
3. **Use semantic keys** — `"label.firstName"` instead of `"text1"`
4. **Support dynamic parameters** — Use `{variable}` syntax for dynamic content
5. **Test RTL layout** — Set `dir="rtl"` and check spacing, icons, and flex directions
6. **Plan for length** — German and Arabic are longer than English; allow 20–30% extra space in UI
7. **Don't translate proper nouns** — Brand names, medication names, clinical terms stay consistent
8. **Coordinate with medical team** — Ensure clinical terminology is accurate in all languages

## Common Issues & Solutions

### Issue: Locale not persisting after reload

**Solution**: Check that the cookie is being set correctly:
```javascript
console.log(document.cookie) // Should show "diabeo_locale=fr"
```

Ensure the cookie path is `/` and doesn't expire immediately.

### Issue: Arabic text is left-aligned instead of right-aligned

**Solution**: Check that `dir="rtl"` is set on the `<html>` element:
```tsx
<html dir={isRtlLocale(locale) ? "rtl" : "ltr"}>
```

### Issue: Flexbox row reversal breaks spacing in RTL

**Solution**: Use logical properties instead of directional margins:
```tsx
// Wrong
<div className="flex gap-4 ml-2">

// Correct
<div className="flex gap-4 ms-2">
```
